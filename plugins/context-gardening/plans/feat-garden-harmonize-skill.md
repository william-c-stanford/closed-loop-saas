# Add /garden:harmonize skill — migrate stray plans and product specs into standard structure

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

When `/garden:init` runs on a repo that already has plans or product specs in non-standard locations — a `plans/` directory generated by another plugin, a root-level `PLANS.md`, spec files under `specs/` or `features/`, `.agent/plans/` from Codex conventions — those files are left untouched. The linter now warns about unlinked plans, but provides no path to fix them.

After this change, a user can run `/garden:harmonize` and the skill will scan the repo, identify stray plan and spec files, assess each one against the ExecPlan and product-spec standards, and interactively offer to convert and relocate the selected files into `docs/execution-plans/active/` and `docs/product-specs/` respectively. It updates `docs/PLANS.md` and `docs/product-specs/index.md` after each migration, and logs what was done.

The observable result: a user with an existing `plans/feat-auth.md` that was generated by the compound-engineering workflows plugin can run `/garden:harmonize`, confirm the file, and have it moved and padded with the required ExecPlan sections — so the linter passes cleanly and the catalogue reflects accurate status.

## Decision Log

- Decision: Implement as a Claude Code skill (SKILL.md), not a linter.
  Rationale: Identifying stray plan/spec files requires semantic judgment — reading content to detect ExecPlan-like structure, classifying lifecycle stage (active vs completed), and drafting missing sections. That judgment is exactly what Claude is good at. A Node.js linter cannot read content semantically.
  Date/Author: 2026-03-07

- Decision: Implement harmonize logic as a standalone skill AND integrate it into init, tend, and weed.
  Rationale: The standalone `/garden:harmonize` handles the explicit "I want to clean up now" case. The init integration handles "I'm setting up a repo that already had plans." The tend integration handles "a new stray file appeared since the last tend run." The weed integration handles "I want one command to clean up all doc problems at once." Each integration point calls the same underlying detection logic but surfaces it in context-appropriate ways (init and tend offer migration; weed offers migration or deletion).
  Date/Author: 2026-03-07

- Decision: Handle both plans and product specs in a single skill.
  Rationale: They have the same workflow pattern (scan → assess → multi-select → convert/move → update index) and are often co-located in the same non-standard directories. Combining them avoids two separate interactive sessions.
  Date/Author: 2026-03-07

- Decision: Add missing ExecPlan sections with placeholder content when converting, rather than requiring the user to add them manually.
  Rationale: The point of harmonize is to make linter errors go away immediately. A plan that is moved but still missing `## Progress` will still fail `plans-active-structure`. The skill should produce a passing file.
  Date/Author: 2026-03-07

- Decision: Replace original file location with a forwarding stub, not a silent delete.
  Rationale: Other tools, agents, or humans may still reference the old path. A one-line stub ("this plan has moved to docs/execution-plans/active/...") prevents confusion without breaking references.
  Date/Author: 2026-03-07

- Decision: Classify lifecycle stage (active vs completed) by reading content, not by asking.
  Rationale: Most plan files contain progress indicators or outcome summaries that make stage obvious. Asking the user to classify each file individually would be tedious for repos with many plans. The skill should infer and show its reasoning, then let the user override.
  Date/Author: 2026-03-07

## Context and Orientation

This repo is a Claude Code plugin. All agent intelligence lives in SKILL.md files under `skills/<name>/SKILL.md`. The skill is the entire implementation — there is no backing code. Key patterns to follow:

- `skills/weed/SKILL.md` — the canonical pattern for interactive discovery + AskUserQuestion multi-select + per-file action + gardening-log append. Harmonize follows this pattern closely.

- `docs/PLANS.md` (scaffolded by init) — the catalogue that harmonize must update after migrating plans.
- `docs/product-specs/index.md` (scaffolded by init) — the catalogue that harmonize must update after migrating specs.

A "stray" plan file is any `.md` file that looks like a plan (based on location or content) but is not already under `docs/execution-plans/`. A "stray" spec file is any `.md` file that looks like a product spec but is not under `docs/product-specs/`.

The ExecPlan required sections are: `## Progress`, `## Decision Log`, `## Outcomes & Retrospective`. If any are missing, the skill adds them with minimal placeholder content so the linter passes.

The plugin.json registers skills; it must be updated when a new skill directory is added. The description field in `plugin.json` is what Claude Code shows in the command palette.


## Plan of Work

The work has four deliverables: the standalone `skills/harmonize/SKILL.md`, and edits to the init, tend, and weed skills. All four share the same underlying detection logic (Steps 2–4 of harmonize), which is described once in harmonize's SKILL.md and referenced by the other three.

**Discovery phase (Steps 1–3):** The skill searches the repo for candidate files in known non-standard locations and by content signals. Known locations to scan: `plans/`, `specs/`, `features/`, `.agent/plans/`, `.agent/specs/`, `PLANS.md` at root, any `*.plan.md` or `*-plan.md` files at root, and the `docs/execution-plans/` subdirectory itself (to find any files that are already in place but missing required sections). Content signals for a plan: presence of `## Progress` checkboxes (`- [ ]`), `## Milestones`, `## Acceptance Criteria`, `## Decision Log`, `## Implementation`. Content signals for a product spec: presence of `## User Story`, `## Problem Statement`, `## Acceptance Criteria` without plan-style checkboxes, `## Proposed Solution`.

**Assessment phase (Step 4):** For each candidate, the skill classifies it: type (plan vs spec), lifecycle stage (active vs completed — inferred from content: if `## Outcomes & Retrospective` has real content and Progress shows all items checked, classify as completed; otherwise active), and gap list (which required sections are missing for plans; whether the spec has a problem statement and acceptance criteria for specs).

**Presentation phase (Step 5):** The skill groups findings into two multi-select lists — one for plans, one for specs — and uses AskUserQuestion to let the user select which files to migrate. Each option shows: current path, inferred type and stage, gap list, and line count. A "skip all" option is always available.

**Migration phase (Steps 6–7):** For each confirmed file the skill: (a) determines destination path (`docs/execution-plans/active/` or `docs/execution-plans/completed/` for plans, `docs/product-specs/` for specs); (b) reads the file content; (c) adds any missing required sections with placeholder content appended to the end; (d) writes the file to the destination using the Write tool; (e) replaces the original with a one-line forwarding stub; (f) updates the catalogue (`docs/PLANS.md` for plans, `docs/product-specs/index.md` for specs) by inserting a new row in the appropriate table.

**Logging phase (Step 8):** Appends a dated entry to `docs/gardening-log.md` listing what was moved, what was added as stubs, and what was skipped.

The `plugin.json` registration is a one-line addition to the existing skills array.

## Idempotence and Recovery

Running harmonize twice on the same repo is safe. On the second run, the files are already in `docs/execution-plans/` and `docs/product-specs/` and will not appear as candidates (they are already in the standard locations). The forwarding stubs at the original paths will be found but filtered out (they are too short and contain a forwarding marker). If a migration is interrupted partway through, re-running harmonize will find the partially-migrated file at its destination and recognize it is already there; it will skip it.


## Artifacts and Notes

The forwarding stub written to the original location looks like this:

    # [Moved] <original filename>

    This plan has moved to [`docs/execution-plans/active/<filename>`](docs/execution-plans/active/<filename>.md).

    It was migrated by `/garden:harmonize` on YYYY-MM-DD.

For specs the stub is analogous, pointing to `docs/product-specs/`.

When appending missing ExecPlan sections to a plan file, use these minimal placeholders:

    ## Progress

    _No progress entries yet. Add checkboxes as work begins._

    ## Decision Log

    _No decisions recorded yet._

    ## Outcomes & Retrospective

    _Populated at completion._

These satisfy the linter's section-presence check (`check-plans.js` looks for the heading string, not content beneath it) while being honest that they are placeholders.

When updating `docs/PLANS.md`, insert a new row into the Active or Completed table. The row format is:

    | [<title>](execution-plans/active/<filename>.md) | Active | <YYYY-MM> | <one-line summary inferred from file content> |

When updating `docs/product-specs/index.md`, insert a new row into the Active Specs table:

    | [<title>](product-specs/<filename>.md) | Draft | <one-line description> |


## Interfaces and Dependencies

`skills/harmonize/SKILL.md` must implement these eight steps precisely:

**Step 1 — Check prerequisites**
Verify `docs/PLANS.md` and `docs/product-specs/index.md` exist. If either is missing, tell the user to run `/garden:init` first and stop. This prevents the skill from running on a repo that has not been initialized.

**Step 2 — Scan for stray plans**
Search these locations for `.md` files that are not already under `docs/execution-plans/`:
- `plans/` directory (any depth)
- `.agent/plans/` directory (any depth)
- Root-level files matching `PLANS.md`, `*-plan.md`, `*.plan.md`
- Any `.md` file outside `docs/` containing two or more of these headings: `## Progress`, `## Milestones`, `## Acceptance Criteria`, `## Decision Log`, `## Implementation Plan`

Exclude: `docs/`, `node_modules/`, `.git/`, `README.md`, and any file whose content is a forwarding stub (contains the string "has moved to").

**Step 3 — Scan for stray product specs**
Search these locations for `.md` files that are not already under `docs/product-specs/`:
- `specs/` directory (any depth)
- `features/` directory (any depth)
- `.agent/specs/` directory (any depth)
- Any `.md` file outside `docs/` containing two or more of these headings: `## User Story`, `## Problem Statement`, `## Acceptance Criteria`, `## Proposed Solution`, `## Overview` combined with `## Acceptance Criteria`

Exclude same list as Step 2. Also exclude files already matched as stray plans (a file can only be one type).

**Step 4 — Classify each candidate**
For each stray plan: read the file, determine lifecycle stage (active if Progress section has unchecked items `- [ ]` or no Progress section; completed if all checkboxes checked or Outcomes section has substantive content), list missing required sections.
For each stray spec: read the file, note whether it has a problem statement and acceptance criteria.

**Step 5 — Present to user**
If no candidates found: report "No stray plans or product specs found. Your structure is already harmonized." and stop.
Otherwise: use AskUserQuestion with multiSelect: true to show all candidates. Group plans first, then specs. Each option label: `<current-path> → docs/execution-plans/<stage>/<filename>.md  [plan, <stage>, missing: <sections or "none">]`. For specs: `<current-path> → docs/product-specs/<filename>.md  [spec]`.

**Step 6 — Migrate selected plans**
For each selected plan:
1. Read content
2. Append any missing required sections with placeholder content (see Artifacts)
3. Write to `docs/execution-plans/<active|completed>/<filename>.md`
4. Write forwarding stub to original path
5. Insert row in `docs/PLANS.md` under the appropriate table

**Step 7 — Migrate selected specs**
For each selected spec:
1. Read content
2. Write to `docs/product-specs/<filename>.md`
3. Write forwarding stub to original path
4. Insert row in `docs/product-specs/index.md` Active Specs table

**Step 8 — Log and report**
Append a dated entry to `docs/gardening-log.md`. Report to the user: how many files migrated, where they landed, and that they should run `/garden:status` to confirm the linter is clean.
